Policy Configuration Guide

How to translate a ROPA requirements document into a consent policy JSON.

Config Reference
Start Here: The ROPA Document

Before writing any JSON, the DPO hands you a plain-text ROPA requirements document. It captures every data processing activity, the processors authorised for each, retention periods, and cross-border transfers. Your job as the engineer is to translate this into the policy JSON that the system ingests.

The walkthrough below uses examples/ropa/retail/apna_mart_customer_v1.txt as the source and builds toward examples/policy/retail/apna_mart_customer_v1.json. Every section of the ROPA document maps to a specific JSON field — nothing is invented.

examples/ropa/retail/apna_mart_customer_v1.txt
PROCESSING ACTIVITY Activity Name : apna_mart_customer_v1 Legal Basis : CONSENT Purpose : - Account & Loyalty Management - Order Fulfillment & Delivery - Personalized Recommendations DATA SUBJECT CATEGORIES - customer DATA CATEGORIES (9 total) full_name mobile_number email_address date_of_birth delivery_address payment_details purchase_history shopping_preferences browsing_history DATA PROCESSORS (4 declared) Razorpay IN Payment Gateway dpo@razorpay.com Delhivery IN Logistics & Last-Mile Delivery privacy@delhivery.com MoEngage IN Customer Engagement & Push privacy@moengage.com GoogleAnalytics US Website & App Analytics google-privacy@google.com PURPOSE-PROCESSOR MAPPING purpose_account_loyalty → MoEngage purpose_order_fulfillment → Delhivery, Razorpay purpose_personalized_offers → MoEngage, GoogleAnalytics CROSS-BORDER TRANSFERS US Standard Contractual Clauses (SCC) GoogleAnalytics RETENTION Period : 2555 days (approx. 7.0 years) Start Event : CESSATION

This is what the DPO gives you. The steps below show how each section becomes JSON.

Step 1 Turn each purpose into a JSON entry

Each purpose listed under Purpose in the ROPA document becomes one object in data_processing_purposes. The id you assign here is the purpose_id used everywhere in the system — consent records, ROPA entries, and the validate_consent API. Use a consistent snake_case naming convention that matches the purpose-processor mapping table in the ROPA document.

ROPA: Purpose list
- Account & Loyalty Management - Order Fulfillment & Delivery - Personalized Recommendations
→
JSON: data_processing_purposes
[ { "id": "purpose_account_loyalty", ... }, { "id": "purpose_order_fulfillment", ... }, { "id": "purpose_personalized_offers", ... } ]

A fully expanded purpose entry looks like this:

Single purpose — full structure
{ "id": "purpose_order_fulfillment", "name": "Order Fulfillment & Delivery", "description": "To process purchases, manage payments, and deliver to your doorstep.", "legal_basis": "Contractual Necessity", "data_categories_involved": ["delivery_address", "payment_details", "purchase_history"], "recipients_or_third_parties": ["Delhivery", "Razorpay"], // filled in Step 3 "retention_policy": "Statutory retention of 7 years for tax and audit purposes.", "retention_duration_value": 7, "retention_duration_unit": "YEARS", "retention_start_event": "CESSATION", "is_mandatory_for_service": true, "is_sensitive": false }
FieldAllowed valuesNotes
retention_duration_unitYEARS, MONTHS, DAYSThe ROPA doc gives days (2555). Convert to the most readable unit — here 7 YEARS.
retention_start_eventCOLLECTION, CESSATIONCESSATION = clock starts when the service relationship ends. Use for transactional data. Use COLLECTION for analytics or marketing data.
is_mandatory_for_servicetrue, falseSet true for purposes without which the service cannot function (order, payment). The UI will lock the toggle for these.
is_sensitivetrue, falseSet true for financial, biometric, health, or government-ID data.
Step 2 Build the data dictionary

Every data category listed in the ROPA document needs a corresponding entry in data_categories_details. The id here must match exactly what you used in each purpose's data_categories_involved. Write a plain-language description for each — this is what the Data Principal reads in the consent notice.

ROPA: Data Categories
full_name mobile_number email_address delivery_address payment_details purchase_history ...
→
JSON: data_categories_details
[ { "id": "full_name", "name": "Full Name", "description": "Customer name for billing and delivery." }, { "id": "payment_details", "name": "Payment Details", "description": "Card or UPI details for order processing." }, ... ]
Step 3 Build the processor registry

The DATA PROCESSORS table in the ROPA document maps directly to the top-level processors array. Include every processor listed — this is the master registry the ROPA module reads to populate the compliance register.

ROPA: Data Processors
Razorpay IN Payment Gateway dpo@razorpay.com Delhivery IN Logistics & Delivery privacy@delhivery.com MoEngage IN Customer Engagement privacy@moengage.com GoogleAna US Website Analytics google-privacy@google.com
→
JSON: processors
[ { "name": "Razorpay", "country": "IN", "role": "Payment Gateway", "contact": "dpo@razorpay.com" }, { "name": "Delhivery", "country": "IN", "role": "Logistics & Last-Mile Delivery", "contact": "privacy@delhivery.com" }, { "name": "MoEngage", "country": "IN", "role": "Customer Engagement & Push", "contact": "privacy@moengage.com" }, { "name": "GoogleAnalytics", "country": "US", "role": "Website & App Analytics", "contact": "google-privacy@google.com" } ]
Step 4 Map processors to each purpose

This is the most important nuance. Declaring a processor in the registry (Step 3) does not authorise it for every purpose. The ROPA document's PURPOSE-PROCESSOR MAPPING table tells you exactly which processors go into each purpose's recipients_or_third_parties field.

Why this matters: A Data Processor calling validate_consent only receives a granted response for the specific purposes they are authorised for. Razorpay cannot process data for purpose_personalized_offers even if the Data Principal has consented to it, because Razorpay is not listed under that purpose.
ROPA: Purpose-Processor Mapping
purpose_account_loyalty → MoEngage purpose_order_fulfillment → Delhivery, Razorpay purpose_personalized_offers → MoEngage, GoogleAnalytics
→
JSON: recipients_or_third_parties per purpose
{ "id": "purpose_account_loyalty", "recipients_or_third_parties": ["MoEngage"], ... }, { "id": "purpose_order_fulfillment", "recipients_or_third_parties": ["Delhivery", "Razorpay"], ... }, { "id": "purpose_personalized_offers", "recipients_or_third_parties": ["MoEngage", "GoogleAnalytics"], ... }

If a purpose is processed entirely in-house with no third party, set the field to an empty array: "recipients_or_third_parties": [].

Step 5 Declare cross-border transfers

Any processor in a country other than India (country code not IN) must be declared in cross_border_transfers with the applicable data protection safeguard. The ROPA document lists these explicitly.

ROPA: Cross-Border Transfers
US Standard Contractual Clauses (SCC) GoogleAnalytics
→
JSON: cross_border_transfers
[ { "destination_country": "US", "safeguard": "Standard Contractual Clauses (SCC)", "processor": "GoogleAnalytics" } ]
Step 6 Add regional languages

Once the English content is complete, duplicate the entire "en" block under a new language key ("ta", "hi", "kn", etc.) and translate only the display text. The id values for purposes and data categories must remain identical across all language variants.

Top-level structure with two languages
{ "en": { "title": "Apna Mart Retail - Data & Privacy Policy", "data_processing_purposes": [ { "id": "purpose_account_loyalty", "name": "Account & Loyalty Management", ... } ], ... }, "ta": { "title": "அப்னா மார்ட் - தரவு மற்றும் தனியுரிமைக் கொள்கை", "data_processing_purposes": [ { "id": "purpose_account_loyalty", "name": "கணக்கு மற்றும் விசுவாசம் மேலாண்மை", ... } ], // id is identical — only name and description change ... } }
