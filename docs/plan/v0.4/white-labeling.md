# Partner White-Labeling (Brand Name Configuration)
**TSI DPDP Consent Management System - Design & Implementation Plan**
Version: 0.4 | Status: Planned

---

## 1. What is White-Labeling Here?

Partners and resellers who deploy this CMS for their own customers want the product to carry *their* brand, not "TSI DPDP CMS." White-labeling, as scoped here, means letting an operator set a **single environment variable** - `BRAND_NAME` - at deployment time, and having every user-facing surface (console, login screens, the data-principal rights portal, the evaluator tour, generated report footers) display that name instead of the default.

The name is capped at **12 characters**. That number is not arbitrary: **"TSI DPDP CMS" is itself exactly 12 characters**. The cap is a deliberate drop-in-replacement constraint - it guarantees a partner's brand fits every layout (sidebar widths, title bars, header chrome, report footers) that was designed around the current brand string, with zero redesign risk.

This is purely a **presentation-layer** concern: it does not touch the database schema, business logic, or per-fiduciary configuration (which is already orthogonal - `fiduciaries` rows have no branding columns and don't need any).

---

## 2. Why Do We Need This?

### 2.1 Partner/Reseller Go-to-Market

Resellers evaluating or operating white-labeled deployments for their own client base cannot present a product that visibly says "TSI DPDP CMS" throughout - it undermines their brand relationship with their customers. A one-variable rebrand removes that blocker entirely.

### 2.2 "Consistent Look and Feel" Is the Actual Requirement

The ask explicitly links the character cap to visual consistency. The brand string today appears in fixed-width UI chrome (console sidebar headers, login card titles, modal headings, report footers). An unconstrained brand name risks overflow, wrapping, or truncation in those exact spots. Matching the cap to the current string's length (12) sidesteps the problem completely - every layout that works today provably continues to work for any compliant partner name.

### 2.3 No Existing Mechanism Covers This

The static `web/` assets are served by Jetty as plain files - there is **no templating engine** (no JSP/Freemarker/Velocity configured in `web/WEB-INF/web.xml`) and **no runtime config-fetch endpoint** the frontend could use to pull a brand string. Backend env-var configuration exists (`SystemConfig`, `JWTUtil`, `DbEncryption`) but nothing wires environment values into HTML output. A new mechanism is required.

## 3. Inventory: Where the Brand Appears Today

A scan of `web/**/*.html` finds four literal brand-token variants, ~56 occurrences across 30+ static files:

| Token | Count | Where it shows up |
|---|---|---|
| `TSI DPDP CMS` | 47 | `<title>` tags; console sidebar/header spans (`web/console/**`); login screens (`web/index.html`, `web/password-reset.html`); the rights portal (`web/rights/index.html`, `web/rights/dashboard.html`); the ROPA report footer ("Generated by..." in `web/console/dpo/ropa.html`); a consent-artifact modal heading; tour copy |
| `TSI Coop` | 5 | `alt="TSI Coop"` on logo `<img>` tags in tour pages, each wrapped in a link to `tsicoop.org` |
| `TSI DPDP` | 3 | Shorter `<title>` variant in two tour pages (`compliance-infographic.html`, `dpdp-wallet.html`) |

Backend Java has no user-facing brand strings to change: reports are CSV-only (no brand text embedded), and no email/notification template references the brand.

## 4. Design Decisions

### 4.1 Runtime Response Rewriting, Not Build-Time Templating

`BRAND_NAME` must be settable "in the env" - i.e., per deployment, without rebuilding the WAR/image. Build-time substitution (Maven resource filtering, baking the string into the artifact) would force a separate build per partner, which defeats the purpose. Instead, the brand is substituted **at serve time**, by a servlet filter that rewrites HTML responses before they reach the client. The ~47+ literal `TSI DPDP CMS` strings in HTML stay exactly as they are - they remain the default text, and only get rewritten when a custom brand is actually configured.

### 4.2 Fail Fast on an Oversized Brand Name

If `BRAND_NAME` is set but exceeds 12 characters, the application **refuses to start**, throwing `IllegalStateException` with a clear message. This mirrors the existing validation style for `JWT_SECRET` (`JWTUtil.loadSecretKey()`) and `DB_ENCRYPTION_KEY` (`DbEncryption`'s static initializer) - both fail fast at class-load if their env var is missing or invalid. Forcing the operator to fix a misconfiguration before going live is preferable to shipping a broken or overflowing layout to end users.

### 4.3 Scope Includes the Evaluator Tour

The tour (`web/tour/**`) is in scope, including its five `tsicoop.org` logo+link blocks (`<a href="https://tsicoop.org"><img src="https://tsicoop.org/logo.svg" alt="TSI Coop">`). For a true white-label deployment, an outbound link/logo pointing at the upstream organization's site inside a partner's own evaluation experience is precisely the kind of leak this feature exists to close.

### 4.4 Logos Stay Text-Only - No `BRAND_LOGO_URL` Knob

Rather than adding a second configuration variable for a partner-supplied logo image, the five `tsicoop.org` logo+link blocks are converted to plain text wordmark spans - the same treatment already applied to `web/rights/index.html` and `web/rights/dashboard.html` in this release (replacing the linked TSI Coop `<img>` with a styled `<span>TSI DPDP CMS</span>`). Rationale: an arbitrary partner-supplied image has arbitrary dimensions and aspect ratio - exactly the kind of variability the 12-character cap exists to eliminate. Keeping the mechanism text-only and singular (one env var, one substitution path) preserves the "guaranteed consistent look and feel" property end-to-end, and removes the outbound link to the upstream site as a side effect.

### 4.5 Ordered, Longest-Match-First Substitution

Because `TSI DPDP CMS` contains `TSI DPDP` as a prefix, replacing the shorter token first would corrupt the longer one (leaving a mangled `{Brand} CMS`). Substitution must run in a fixed order - longest first:

```
"TSI DPDP CMS"  →  {brand}
"TSI Coop"      →  {brand}
"TSI DPDP"      →  {brand}
"TSI CMS"       →  {brand}
```

`TSI Coop`, `TSI DPDP`, and `TSI CMS` don't overlap each other, so only `TSI DPDP CMS` needs to precede the rest. Matching is case-sensitive literal text matching, so it cannot collide with lowercase code-level identifiers such as the `--tsi-primary` CSS variable or the `tsi_internal` Docker network name.

## 5. Architecture

### 5.1 `BrandConfig`

New class `org.tsicoop.dpdpcms.framework.BrandConfig`, mirroring `DbEncryption`'s static-initializer pattern:

- Reads `BRAND_NAME` once at class-load.
  - Unset or blank → brand = `"TSI DPDP CMS"` (the default; no rewriting occurs).
  - Set and ≤ 12 characters → brand = the configured value.
  - Set and > 12 characters → `throw new IllegalStateException("BRAND_NAME must be at most 12 characters (got N: '...')")`. The application does not start.
- Exposes `BrandConfig.name()` (the active brand string) and `BrandConfig.isCustomized()` (`true` only when a non-default brand is active - lets the filter take a zero-cost fast path in the common case).
- Owns the ordered replacement table described in §4.5.

### 5.2 `BrandingFilter`

New `Filter` in the same package, registered in `web/WEB-INF/web.xml` next to the existing `InterceptingFilter` (whose `<filter>`/`<filter-mapping>` block at lines 27-37 is the template to follow). Mapped across the full HTML surface - welcome file `/`, `*.html`, `/console/*`, `/rights/*`, `/tour/*` - since the tour is in scope.

- **Fast path**: when `!BrandConfig.isCustomized()`, the filter calls `chain.doFilter()` directly with no buffering - deployments that don't set `BRAND_NAME` see no behavioral or performance change at all.
- **Custom-brand path**: wraps the response in a buffering `HttpServletResponseWrapper` (a small `CharResponseWrapper` capturing `getWriter()`/`getOutputStream()` into an in-memory buffer), allows the chain to render normally, runs the ordered replacements from `BrandConfig` over the buffered HTML, recalculates `Content-Length`, and writes the rewritten bytes to the real response stream.

### 5.3 Request Flow (custom brand configured)

```
Client GET /console/dpo/dashboard.html
   → BrandingFilter (isCustomized() == true)
       → chain.doFilter(request, wrappedResponse)
           → Jetty serves the static HTML into the buffer
       → BrandingFilter reads buffered HTML
       → replaces "TSI DPDP CMS" / "TSI Coop" / "TSI DPDP" / "TSI CMS" → {BRAND_NAME}, longest-match-first
       → recomputes Content-Length, writes rewritten bytes to the real response
   ← Client receives HTML branded as {BRAND_NAME}
```

## 6. Implementation

### Step 1 - `BrandConfig`
Add `src/org/tsicoop/dpdpcms/framework/BrandConfig.java`: env var read, 12-char fail-fast validation (`IllegalStateException`, matching `JWTUtil`/`DbEncryption` style), `name()`/`isCustomized()` accessors, and the ordered variant→brand replacement table from §4.5.

### Step 2 - `BrandingFilter` + response wrapper
Add `src/org/tsicoop/dpdpcms/framework/BrandingFilter.java` implementing `Filter`, plus a small buffering `CharResponseWrapper` (inner or companion class) extending `HttpServletResponseWrapper`. Fast-path skip when `!BrandConfig.isCustomized()`; otherwise buffer → substitute → recompute `Content-Length` → flush, as described in §5.2.

### Step 3 - Filter registration
Edit `web/WEB-INF/web.xml`: add a `<filter>`/`<filter-mapping>` block for `BrandingFilter`, following the existing `InterceptingFilter` registration (lines 27-37) as the template. Map it over the static HTML surface: welcome file, `*.html`, `/console/*`, `/rights/*`, `/tour/*`.

### Step 4 - Tour logo cleanup
Edit the five tour pages containing `tsicoop.org` logo+link blocks - `web/tour/index.html`, `consent-collector.html`, `consent-verifier.html`, `system-integration.html`, `user-dashboard.html`. Replace each `<a href="https://tsicoop.org" ...><img src="https://tsicoop.org/logo.svg" alt="TSI Coop" .../></a>` with a styled `<span>TSI DPDP CMS</span>` text wordmark, matching the conversion already done for `web/rights/index.html` / `dashboard.html` in this release.

### Step 5 - Documentation & deployment config
- Add `BRAND_NAME` to `.example`, `server/.example`, and the `jetty_app.environment` block in `docker-compose.yml`, following the pattern of existing optional vars like `ALLOWED_ORIGINS` / `TSI_KEYSTORE_ALIAS` - with an inline comment noting the 12-character hard limit and the fail-fast startup behavior.
- Add a short section to `README.md` (near the Implementation Guide or Production Deployment section) describing the white-labeling capability for partners/resellers.

## 8. Verification

### 8.1 Validation Behavior
- `BRAND_NAME` unset → app starts normally; every page still shows `TSI DPDP CMS`; filter takes the fast path (no buffering).
- `BRAND_NAME=Acme Privacy` (12 characters, the maximum) → app starts; brand is substituted everywhere.
- `BRAND_NAME=Consent Solutions Inc` (> 12 characters) → app refuses to start; logs show a clear `IllegalStateException` naming the offending value and the limit.

### 8.2 End-to-End Browser Verification
With a valid custom brand configured, load and screenshot (headless Chrome, the same technique used to verify the rights-portal branding change earlier in this release):
- Console login (`web/index.html`) and an authenticated console page (`web/console/dpo/dashboard.html`)
- Rights portal login and dashboard (`web/rights/index.html`, `dashboard.html`)
- A tour page (e.g. `web/tour/index.html`)
- A generated ROPA report (footer "Generated by ...")

Confirm: every surface shows the partner brand consistently; no leftover `TSI DPDP CMS` / `TSI Coop` / `TSI DPDP` / `TSI CMS` fragments remain; no broken layout, truncation, or wrapping; no outbound `tsicoop.org` links remain in the tour; `curl -I` shows a correct, recalculated `Content-Length` (no truncated/garbled responses).

## 9. Planned Enhancements (Out of Scope for This Pass)

- Per-fiduciary brand override (distinct from the platform-level partner brand) - would require a `fiduciaries` schema change and is independent of this work.
- A configurable brand color/theme accent (currently fixed via the `--tsi-primary` CSS variable) - would need its own design pass on the 12-char-style consistency guarantee equivalent for color values.
